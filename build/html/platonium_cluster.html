Parallel Computing¶

PlatoSim3 as it is (in serial) will need to be submitted with the calculations as single-node jobs, whereas in parallel, multiple nodes can be put in use. We will here start with the simplest HPC usage example and gradually build ontop with advancing examples. All examples presented here can be found in $PLATO_PROJECT_HOME/python/examples/HPC/.

Attention

PlatoSim do not support GPU parallelisation, only CPU parallelisation.

Job script generator¶

To easily prepare larger simulations that needs to run on a computing cluster, we here developed the script called slurmsim. This script will generate the following files:

  • run.pbs : Job script to SLURM

  • data.pbs : Parameterisation file for array-like jobs

Seen here, before running the script with input arguments, you’ll need to customize the user defined parameters within the script. Most importantly, the timePerImage and memoryPerImage needs to be determine accurately (e.g. from a smaller simulation of the eaxct setup) to ensure that the automatic estimation of resources (nodes, cores, memory) are rightfully calculated by the script. The automation ensures that the memory equation (ppn * pmem < nmax - 8 GB) is satisfied together with an optimized use cores per node(s). However, notice the scheduler might choose otherwise e.g., if 2 nodes x 24 cores are specified, the scheduler will most likely try to fill up the first node completely, meaning for a 36 core-node, all 36 cores will be used on the first node and the remaining 12 cores on the second node. Running makeSimQuarter.py (with input arguments) will create a textfile with a line for each of the (groups x telescopes x quarters) observational configurations and a job script to execute these. If the automatic generated job resources given in the simQuarter.pbs doesn’t cut it, the job resources can safely be modified after need.

As a small example of an realistic simulation, assume that we only want to simulate small stellar masks (imagettes) for a sample of stars. The simplest simulation is obviously to select only one group, one telescope, and one quarter: python makeSimQuarter.py 1 1 1. One quarter corresponds to (60 x 60 x 24 x 88) s / 25 s = 304,128 exposures. If one simulation (meaning one imagette of each star) is 50 kb and takes 1 s, then a quarter simulation is 432 Mb and takes 2h 24min using 36 node-cores. Try play around with different configurations and notice that makeSimQuarter.py automatically provides you the calculation just shown - e.g., the simplest simulation just shown and the default simulation prints the following to bash

Preparing   1 simulations with resources: nodes=1, cores=36, pmem=432mb, walltime=2:24:00
Preparing 192 simulations with resources: nodes=6, cores=32, pmem=81mb,  walltime=0:27:00

Worker environment¶

Submitting the job script is simply done by:

wsub -master -batch <job_script.pbs> -data <data_filename.txt>

We here use the flag -master to specify that we want all node-cores to run the simulations, otherwise one of them will simply idle and only handle the job scheduling (which might be handy sometimes).

Parallelise varsim¶

We now arrive to the purpose of the file data_hpc_starcat**.txt which automatically is generated by picsim. This ascii file contains a parameterized list of the bulk paramters of the stellar PIC targets (starID, \(T_{\rm eff}\), \(M\), \(R\)) which for the purpose of parallel computing can be given as input to any embarassingly parallel frameworks such as the popular worker framework. The worker quick start gives a good introduction to the usage of this framework, but notice similar frameworks can do eaxctly the same, and they might be more useful for your cluster setup.

When scheduling a job with worker it will automatically estimate the number of single job items (i.e. the number of rows in your data_hpc_starcat**.txt) you have and try to schedule your job(s) in an optimal way. Assuming that you understand the general concept of a job script (as described in Job scripts) the following shows an example of running varsim in parallel:

#!/bin/bash

#PBS -N <job_name>
#PBS -A <credit_account>
#PBS -l nodes=1:ppn=36
#PBS -l pmem=5gb
#PBS -l walltime=02:00:00

cd $PBS_O_WORKDIR

# Load paths
POETRY=$DATA/poetry/
export POETRY
PLATONIUM=$DATA/PLATOnium
export PLATONIUM
SIMDIR=$PLATONIUM/platonium/var
export SIMDIR
TEMDIR=$SCRATCH_NODE
export TEMDIR
OUTDIR=$SCRATCH/project/varsource
export OUTDIR
PYTHONPATH=$POETRY$PLATONIUM:$SIMDIR:$OUTDIR
export PYTHONPATH

# Activaty Python environment
source $POETRY

# Run varsim and store temporary on node
python $SIMDIR/varsim --star_params $M $R $Teff 4.5 0.0 --xsource $xsource --quarter 1-8 -o $TEMDIR/varsource_${ID}.txt

# Move data to storage location
mv $TEMDIR/varsource_${ID}.txt $OUTDIR

Starting from the top we here use 1 node with 36 cores and requesting the maximum memory RAM (and storage for the current thin nodes) being 5 Gb (i.e. 180 Gb out of which 189 Gb is available). Next we make sure to make all path environments variables globally availble for any given cluster node that our job might run on. An important notice is that we use $SCRATCH_NODE, being the node’s own storage memory, to temporary store the data since saving it directly to the storage location $SCRATC would introduce some overhead due to slower I/O handling. Lastly, simply summit the job as seen above.

Parallelise platonium¶

#!/bin/bash

#PBS -N job_test1
#PBS -A <credit_account>
#PBS -l nodes=1:ppn=22
#PBS -l pmem=8gb
#PBS -l walltime=30:00:00

cd $PBS_O_WORKDIR

# Load paths
POETRY=$DATA/poetry/
export POETRY
PLATO_PROJECT_HOME=$DATA/PlatoSim3
export PLATO_PROJECT_HOME
PLATO_WORKDIR=$DATA/workdir
export PLATO_WORKDIR
PLATONIUM=$DATA/platonium
export PLATONIUM
PLATO=$DATA/plato
export PLATO
SIMDIR=$PLATONIUM/platonium
export SIMDIR
VARDIR=$SCRATCH/platonium/varsource
export VARDIR
TEMDIR=$SCRATCH_NODE
export TEMDIR
OUTDIR=$SCRATCH/platonium/output
export OUTDIR
PYTHONPATH=:$PLATO_PROJECT_HOME/python:$PLATO_WORKDIR:$PLATONIUM:$PLATO:$SIMDIR:$VARDIR:$TEMDIR:$OUTDIR
export PYTHONPATH

# Load python environment
source $POETRY

# Make sure that stars is 9 digits (L1 pipeline requirement)
starID=$(printf "%09d" $star)

# Run PLATOnium
python $SIMDIR/platonium $starID $group $camera $quarter --project test1 -o $TEMDIR --compress -v 0

# Move data from the scrath node to output
mkdir -p $OUTDIR/$starID
zipfile=$TEMDIR/reduced/P1/$starID/${starID}_Ncam${group}.${camera}_Q${quarter}.zip
if [ -f $zipfile ]; then
    mv $zipfile $OUTDIR/$starID
fi
