Parallel Computing
==================

PlatoSim3 as it is (in serial) will need to be submitted with the calculations as single-node jobs, whereas in parallel, multiple nodes can be put in use. We will here start with the simplest HPC usage example and gradually build ontop with advancing examples. All examples presented here can be found in ``$PLATO_PROJECT_HOME/python/examples/HPC/``.

.. attention::

   PlatoSim do **not** support GPU parallelisation, only CPU parallelisation.

slurmsim
--------

**Job script generator for platonium**

To easily prepare larger batches of simulations with ``platonium`` that needs to run on a computing cluster, we here developed the script called ``slurmsim``.

**Usage function:** To get an overview of its usage simply type:

.. code-block::

   slurmsim -h

Running ``slurmsim`` will generate the following files:

  - ``run.slurm`` : Job script to SLURM
  - ``data.txt``  : Parameterisation file

We will come to the job script in a moment. The a parameterisation file will have row for each of the (group, camera, quarter) observational configurations that is possible to parse to ``platonium``. Don't worry if this file contains camera (group, camera, quarter) configurations that cannot be simulated, ``platonium`` will automatically skip stars that are not observable. 
   
**General usage:** It is possible to parse your input project path in the two following ways:
   
.. code-block::

   slurmsim 100 --project <project_name>
   slurmsim 100 --project <project_name> [--nodes <value> --cores <value> --mem <value>]
   slurmsim 100 --project <project_name> [--tsim <value> --msim <value>]
    
See here the a few arguments can be parsed:

* ``--project``: to parse the project location
* ``--nodes``: number of nodes to use (default: 1)
* ``--cores``: number of CPUs per core (default: 36)
* ``--mem``: RAM memory per CPU (default: 1 GB)
* ``--tsim``: execution time of a single simulation (default: 600s)
* ``--msim``: RAM memory of a single simulation (default: 200 Mb)

Using these input parameter, the script will automatically ensure that the RAM memory equation (``cores * mem < nmax - 8 GB``) is satisfied, and that the use cores per node(s) is optimised. However, notice the scheduler might choose otherwise e.g., if 2 nodes x 24 cores are specified, the scheduler will most likely try to fill up the first node completely, meaning for a 36 core-node, all 36 cores will be used on the first node and the remaining 12 cores on the second node. Try play around with different configurations and notice that ``slurmsim`` automatically print the needed resources to bash, e.g.:

.. code-block:: shell
		
   Preparing 192 simulations with resources: nodes=6, cores=32, pmem=81mb, walltime=0:27:00

If the automatic generated job resources given in ``run.slurm`` doesn't cut it, the job resources can safely be modified after need. It is recommend always to make a test run on your cluster using all CPUs of a single node. In this way you can easily extrapolate the resources needed for your job, since nodes (almost) run independently of each other on a cluster.

Worker environment
------------------

To submit the following job scripts we use the popular `worker <https://worker.readthedocs.io/en/latest/>`_ framework. The `worker quick start <https://vlaams-supercomputing-centrum-vscdocumentation.readthedocs-hosted.com/en/latest/jobs/worker_framework.html>`_ gives a good introduction to the usage of this framework, but notice similar frameworks can do eaxctly the same, and they might be more useful for your cluster setup.

Submitting the job script is simply done by:

.. code-block:: shell
		
   wsub -master -sbatch run.slurm -data data.txt

We here use the flag ``-master`` to specify that we want all CPUs to run the simulations, otherwise one of them will simply idle and only handle the job scheduling (which might be handy sometimes).

.. _hpc_tutorial_varsim:
    
Parallelise varsim
------------------

We now arrive to the purpose of the file ``data_hpc_starcat**.txt`` which automatically is generated by ``picsim``. This ascii file contains a parameterized list of the bulk paramters of the stellar PIC targets (starID, :math:`T_{\rm eff}`, :math:`M`, :math:`R`) which for the purpose of parallel computing can be given as input to any *embarassingly parallel* frameworks such as ``worker``.

When scheduling a job with worker it will automatically estimate the number of single job items (i.e. the number of rows in your ``data_hpc_starcat**.txt``) you have and try to schedule your job(s) in an optimal way. Assuming that you understand the general concept of a job script (as described in :ref:`hpc_intro_job`) the following shows an example of running ``varsim`` in parallel:

.. code-block:: shell

   #!/bin/bash

   #PBS -N <job_name>
   #PBS -A <credit_account>
   #PBS -l nodes=1:ppn=36
   #PBS -l pmem=5gb
   #PBS -l walltime=02:00:00

   cd $PBS_O_WORKDIR

   # Load paths
   POETRY=$DATA/poetry/
   export POETRY
   PLATONIUM=$DATA/PLATOnium
   export PLATONIUM
   SIMDIR=$PLATONIUM/platonium/var
   export SIMDIR
   TEMDIR=$SCRATCH_NODE
   export TEMDIR
   OUTDIR=$SCRATCH/project/varsource
   export OUTDIR
   PYTHONPATH=$POETRY$PLATONIUM:$SIMDIR:$OUTDIR
   export PYTHONPATH

   # Activaty Python environment
   source $POETRY

   # Run varsim and store temporary on node
   python $SIMDIR/varsim --star_params $M $R $Teff 4.5 0.0 --xsource $xsource --quarter 1-8 -o $TEMDIR/varsource_${ID}.txt

   # Move data to storage location
   mv $TEMDIR/varsource_${ID}.txt $OUTDIR 

Starting from the top we here use 1 node with 36 cores and requesting the maximum memory RAM (and storage for the current thin nodes) being 5 Gb (i.e. 180 Gb out of which 189 Gb is available). Next we make sure to make all path environments variables globally availble for any given cluster node that our job might run on. An important notice is that we use ``$SCRATCH_NODE``, being the node's own storage memory, to temporary store the data since saving it directly to the storage location ``$SCRATC`` would introduce some overhead due to slower I/O handling. Lastly, simply summit the job as seen above.

   Simplest parallel job
   ---------------------

   As mentioned before, simulations can be executed either directly from bash or from a python script. Let's illustrate this with the PlatoSim3's default simulation (see **testPlatosim**) where we use the ``inputfiles/inputfile.yaml`` simulating 10 imagettes. Since this simulation can be done on our local machine, a test shows that approximately 1 imagettes/second is simulated and the ``output.hdf5`` file takes up around 10 Mb. Hence, as this job requieres very limited resources and we will use a debugging node. A reasonable job script (without running PlatoSim3) would be 

   .. code-block:: shell

      #!/bin/bash

      #PBS -l nodes=1:ppn=1
      #PBS -l pmem=150mb
      #PBS -l walltime=00:00:30
      #PBS -l qos=debugging
      #PBS -A default_project
      #PBS -N output

      cd $PBS_O_WORKDIR

      module purge
      module restore env-PlatoSim3

      PLATO_PROJECT_HOME=$VSC_DATA/PlatoSim3
      export PLATO_PROJECT_HOME
      PYTHONPATH=$PLATO_PROJECT_HOME/python:$VSC_DATA/python_lib/lib/python3.7/site-packages/
      export PYTHONPATH

   Notice that the ``PLATO_PROJECT_HOME`` and ``PYTHONPATH`` needs to be defined and exported within every job script in order for the PBS system to recognize them. Now running PlatoSim3 using bash is done by adding the following line of code to the job script (``testPlatosim-sh.pbs``)

   .. code-block:: shell

      $PLATO_PROJECT_HOME/build/platosim $PLATO_PROJECT_HOME/inputfiles/input.yaml output.hdf5

   The same simple job script can be utilized using python scripting as seen here (``testPlatosim.py``)

   .. code-block:: shell

      #!/usr/bin/env python3

      import os
      from platosim.simulation import Simulation

      runDir  = os.getcwd()
      simName = 'ouput'
      sim     = Simulation(simName, outputDir=runDir)
      sim["ObservingParameters/NumExposures"] = 10
      simFile = sim.run()

   and insted we add the following line of code to our job script (``testPlatosim-py.pbs``)

   .. code-block:: shell

      python testPlatosim.py



   Enabling the *sequential* parallelism of PlatoSim3
   --------------------------------------------------

   As stated in the PlatoSim3 documentation under `Configuration Parameters <http://ivs-kuleuven.github.io/PlatoSim3/ConfigurationParameters.html>`_, one can take advantage of PlatoSim3's (quasi) parallelism in its source code using the input parameter ``BeginExposureNr``. As ``BeginExposureNr`` defines the number of the first exposure in a sequence, bigger simulations can be chopped-up into smaller simulation chunks and handled in parallel by the scheduler. E.g., if ``BeginExposureNr = 10`` and 100 simulations are choosen, then 10 simulation chunks can be handled in parallel. However, the backward calculation of defining ``BeginExposureNr`` from the number images forming a sequence, or the total number of smaller simulation jobs, are usually desired. Hence, for demonstration a python module called ``runSequential.py``, that uses these formalisms, is available in **simSequential** folder. Running ``python runSequential.py`` (without input arguments) will print the usage of the script, however, be aware that this module needs to launched by the job script called ``jobSequential.pbs``. 

   In order to use these demo scripts you'll first need to edit/add all the relevant informations regarding your simulation under the *Setup simulations* within ``runSequential.py``. Specifically, you should define the total number of exposures, ``Nexps``, of your combined simulation together with the parameter ``N`` that, from the discussing above, is defined as the number of images in one sequence (if ``selectSequence = True``) or the total number of parallel jobs (if ``selectSequence = False``). As an example, say that ``Nexps = 105`` and ``N = 10``. While ``selectSequence = True`` notice in the following that this generate 11 jobs and that the last job only include 5 images

   .. code-block:: shell

      Monitoring (Njobs, Nexp)=(11, 105): jobNr=1, jobExp=1-10
      .
      .
      Monitoring (Njobs, Nexp)=(11, 105): jobNr=11, jobExp=101-105

   whereas if ``selectSequene = False`` you've bootstrapped ``N = Njobs = 10`` and the script will try to divide ``Nexp`` up into 10 equal simulations. In this example, this is clearly not possible (since ``Njobs = 10.5``) and instead the first ``Njobs-1`` jobs will include ``ceil(Njobs) = 11`` images, whereas the modulus of 6 images are placed in the last job as follows 

   .. code-block:: shell

      Monitoring (Njobs, Nexp)=(10, 105): jobNr=1, jobExp=1-11
      .
      .
      Monitoring (Njobs, Nexp)=(10, 105): jobNr=10, jobExp=100-105

   Hence, if by mistake the ``Nexps/N`` is not an integer, this demo automatically takes care of the last job. While the job script is running on the HPC, the above job message is written to your ``<JobName>.o<JobID>`` for you to monitor the job and for extra trouble shooting.

   Finally, it is time to adjust the PBS resources within the job script ``jobSequential.pbs``, and when everything is set up, the job script can be submitted with worker using

   .. code-block:: shell

      module load worker
      wsub -t 1-<Njobs> -batch jobSequential.pbs

   Notice that different from SLURM job arrays, first both numbers in the ranges shown in this section are included in the Torque array calculations and, secondly, the Torque ``qsub`` command (for which ``wsub`` depends on) also takes complex task ID/ranges, e.g., ``qsub -t 1-5,12,17-35``, and hence the min and max array IDs usually used in SLURM are here meaningles. Although not applicable for this demostration, the advantage of this complex array behavior is, e.g., that you can "add" large time jumps to one long simulation, instead of splitting up the simulation into different time domains.

Parallelise platonium
---------------------
    
.. code-block:: shell

   #!/bin/bash

   #PBS -N job_test1
   #PBS -A <credit_account>
   #PBS -l nodes=1:ppn=22
   #PBS -l pmem=8gb
   #PBS -l walltime=30:00:00

   cd $PBS_O_WORKDIR

   # Load paths
   POETRY=$DATA/poetry/
   export POETRY
   PLATO_PROJECT_HOME=$DATA/PlatoSim3
   export PLATO_PROJECT_HOME
   PLATO_WORKDIR=$DATA/workdir
   export PLATO_WORKDIR
   PLATONIUM=$DATA/platonium
   export PLATONIUM
   PLATO=$DATA/plato
   export PLATO
   SIMDIR=$PLATONIUM/platonium
   export SIMDIR
   VARDIR=$SCRATCH/platonium/varsource
   export VARDIR
   TEMDIR=$SCRATCH_NODE
   export TEMDIR
   OUTDIR=$SCRATCH/platonium/output
   export OUTDIR
   PYTHONPATH=:$PLATO_PROJECT_HOME/python:$PLATO_WORKDIR:$PLATONIUM:$PLATO:$SIMDIR:$VARDIR:$TEMDIR:$OUTDIR
   export PYTHONPATH

   # Load python environment
   source $POETRY

   # Make sure that stars is 9 digits (L1 pipeline requirement)
   starID=$(printf "%09d" $star)
   
   # Run PLATOnium
   python $SIMDIR/platonium $starID $group $camera $quarter --project test1 -o $TEMDIR --compress -v 0

   # Move data from the scrath node to output
   mkdir -p $OUTDIR/$starID
   zipfile=$TEMDIR/reduced/P1/$starID/${starID}_Ncam${group}.${camera}_Q${quarter}.zip
   if [ -f $zipfile ]; then
       mv $zipfile $OUTDIR/$starID
   fi
