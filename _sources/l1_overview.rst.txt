In the following we describe how to create representative P1 and P5 sample light curves. The full documentation of how to run PlatoSim imagettes are given in the General :doc:`repo_tutorial`.

The following shows an outline of the reduction steps involved for the **P1 sample** star:

#. Simulate imagettes time series with PlatoSim
#. Simulate microscanning sequence and compute inverted PSFs for each star
#. Pre-process imagette time series (normal- and microscan imagettes)
#. Extract on-ground photometry using PSF fitting

The following shows an outline of the reduction steps involved for the **P5 sample** stars:

#. Simulate imagettes time series with PlatoSim
#. Simulate microscanning sequence and compute inverted PSFs for each star
#. Pre-process imagette time series (normal- and microscan imagettes)
#. Extract on-board photometry using Marchiori+2019 optimal aperture photometry method
#. Apply jitter and drift correction

Microscanning & PSF Inversion
-----------------------------

In flight PLATO will in the interval between each quarter perform microscanning sequence of pre-selected stars. The microscanning sessions are effectively an image time series of controlled spacecraft jitter patterns, where an Arcmedian spiral pattern will be used for the AOCS jitter. Having a sequence of inter-pixel-position shifted images it is possible to estimate a target stars high resolution PSF through the method of PSF inversion.

..
   .. image:: figures/archimedean_spiral.png
      :align: center
      :width: 400
      :alt: Alternative text

Such inversion technique is part of the L1A pipeline and the algorithmic overview consist of

1. ``microscan.py``: PlatoSim is used to perform the microscanning sequence. This is done bu using the jitter file called ``spiral_8Hz_3d.dat`` comprising 430 images of the Archmedian spiral pattern. The jitter pattern include some pointing tolerance. Note that it is best not to subtract the background in PlatoSim, because the module estimates this and subtract it.
2. ``pproc.py``: This is the imagette pre-processing script. 
3. ``invert.sh``: This script calculated the inverted PSFs.
   - It first calls a script called ``extract_contaminats.py`` which providing a <targetID>.txt file that include relative flux and relative positions (x,y) in pixels, will detect and estimates the flux from contaminant stars. Next the executeable ``invert_parabolic_multi`` script is used to find the inverted PSFs using regulazation methods.
   - Next the script invert_parabolic1_multi is called which is the Positive Regulalised Least Square (PRLS) method used for the inversion. Here we can use -p: is to fix resolution of inversion 20 -> 1/20 pixel resolution -> 10 is faster! -r: sampled a spatial resolution as 1/128 pixels compared to (NEEDS yaml: Write pixel, bias, flatfield, starCat, highresPSF)
        
Overall the microscanning take around 40 seconds and inversion last a few minute for a single target star, but considerably longer if contaminants are included as well! Importantly the PSF inversion is stored in *.vec* with B-spline.


Pre-processing
--------------

The first step in the L1 pipeline is running a so-called pre-processing module. This module, called ``pproc.py``, performs the following corrections:


The following mandatory PlatoSim outputs:

- ``WritePixelMaps``
- ``WriteSmearingMaps``
- ``WriteThroughputMaps``
- ``WriteFlatfieldMap`` (Will be calculated in FOV later)
- ``WriteStarPositions``
- ``WriteACS`` (Optional if jitter cadence is below 4 mHz: then use flag ``--high-cadence-displacement``)
- ``WriteDiffusedPSF`` (if MappedFromFile PSFs are used and if diffusion is included)
- ``WriteRotatedPSF``  (if MappedFromFile PSFs are used and if diffusion is turned off)
- ``WriteHighResolutionPSF`` (if analytic non-gaussian PSFs are used)
- ``WriteStarCatalog``
- ``WriteTransmissionEfficiency``

.. note::

   The pre-processing scirpt extract the star IDs from the PlatoSim HDF5 ``StarCatalog/starID`` output file entry. The information given in this data set must be consistent with the catalogue your are working and for which you extract the stars for which you  want to generate simulations.

The pre-processing ``pproc.py`` module only required the output HDF5 and YAML file to generate all the inputs required for all the subsequent processing modules (PSF fitting, photometry extraction and correction, inversion, etc.).

It is important to check the sky background calculation. In-flight the background will be measured on 2000 different locations, and at each exposure we will fit these values using a 2D polynomial which will be then used to infer the sky background at any location in the FoV.
We have not implemented here this complex procedure. Not only ``pproc.py`` needs to correct the imagettes from the background, but also the inversion module (and also the PSF fitting module) needs also to know the level of the sky background at the imagette location, since this value is used to estimate the random noise due to the sky-background (the chi2 which is minimized during the inversion is weighted by the error associated with each pixel value).

On-ground L1 pipeline
---------------------

The strategy for the on-ground photometric extraction is using PSF photometry. This strategy allows higher precision photometry than binary mask extraction, since each star (targets and contaminats) within an imagette can be moduled, which especially is advantagous for PLATO's scarse sampling of the PSF onto the pixel-grid focal plane. This is especially true for crowded regions, however, PSF fitting do also fail when crowding becomes to prominents (e.g. when stars contaminants are within the same pixel as the target star).

The module ``pproc.py`` has the following mandatory PlatoSim outputs:

.. code-block:: shell

   ThroughputMaps
   WriteDiffusedPSF (when analytical MappedFromFile PSFs are used)
   WriteHighResolutionPSF (when analytical non-gaussian PSFs are used)
   WriteTransmissionEfficiency
   WriteStarPositions
   WriteFlatfieldMap
   WriteThroughputMaps
   WriteSmearingMaps



* `lightcurve.py` (with or without mask update)
  * Calling this function twice: to either use real PSF or inverted (last is what we want)

* `jittercorrection.py` 
  * -I: path to the inverse PSF with B-spline
  * --bres 20: B-spline resolution
  * --bsktype: scheme of nodge

* `psffit.py`
  * This include a background option: --no-bg-corr
  * Also takes into account of contaminants but need guesses:
  * See -F, -D, -s, -C, no error for target and contaminant separation as per Gaia is very accurate
  * -P: error of PRNU (0.1%)
  * --fix-relative-pos and --fix-intensity

	      
In-flight L1 pipeline
---------------------


PSF Inversion
-------------

The python script ``invert.py`` now uses this option in order to ignore contaminant stars that are fainter by ``delta_magnitude``. This is 5 mag by default corresponding to a target-to-contaminant flux ratio of 100.

Concerning the resolution of the inverse PSF, it is enough to reduce the value of the parameter associated with the option ``-p`` of ``invert_parabolic1_multi``. Note also that the inversion code ``invert-rls`` can be compiled using ``OPEN_MP`` parallelization directive. In that case a single inversion can be run on several cores. The number of cores can be specified by the system variable ``OMP_NUM_THREADS`` as done in the following example:

.. code-block:: shell
		
   export OMP_NUM_THREADS=1
   invert_parabolic1_multi -o /path/to/output -N 168 -r 128 -p 10 -l 128 -m PRLS -q '+name+'/'+name+'_offsets.txt -Q -u '+  regs + '   -d .  -t continuous  -i ' + name  )


..
   KA on  -> RESULT (regularisation lambda, L1, L2, csx, csy) : 1.883403e-10 0.20870 0.01541 -0.04241 -0.08733
   There is a shift of ~-0.08733 pixel in the Y direction, the shift in the X direction is lower.
   In my opinion this is because when DKA is on, the true position of the star is not the one given by rowPix and colPix.
   As I said some times ago, this will not affect the quality of the L1 klight-curves extracted on the basis of the PSF fitting method (psffit.py) since here the star centroids are free parameters.
   However this shift will affect the jitter/drift correction because the latter needs to know very accurately the star centroid at the beginning of the times-series (typically with ~0.01-0.03 pixel).

   L1 and L2 are respectively the L1 norm and L2 norm
   L1 = integral ( |P_tru - P_inverse|  dx dy)
   L2 = integral ( |P_tru - P_inverse| ^2 dx dy)
   for instance in:
   RESULT (L1,L2,csx,csy) : 1.883403e-10 0.20870 0.01541 -0.04241 -0.08733
   L1 = 0.20870
   L2 = 0.01541
   lambda = 1.883403e-10 (regularization parameter)
   the tow last figures are the centroid shift:
   Dx = -0.04241
   Dy = -0.08733


PSF fitting
-----------

please note that with the option  --fix-relative-pos you can force psffit.py to maintain fixed the distance btw the contaminant stars and the target.
As mentioned by Sébastien Deheuvels it is reasonable to consider this distance as fixed. Note that if the option  --fix-relative-pos, the distance btw a given  contaminant star and the target is only adjusted for the first exposure.
This  can speed-up a little bit the calculation when contaminant stars are present.


Pointing error corrections
--------------------------

The AOCS and TED Actually the 'drift and jitter' correction algo  also requires to infer the displacement of the star during the  time.

This information is currently inferred by pproc.py in two ways:
file <starID>_displacement.dat: the star displacement is computing using the jitter file and the TED file and by applying all the necessary coordinates in order to translate the variation of the plateform attiude (jitter) and TED into star displacement in the focal plane and then in the CCD frame.

File <starID>_displacement2.dat: the star displacement is inferred from the information given by 'row' and 'col'. Thus the star displacement is dx = col(t) - col(t=0) and dy = row - row(t=0).

The advantage of the first method is that we can have the information about the star displacement at high cadence (the cadence of the jitter file: 8Hz when the jitter is turned on and when the OHB time series is used).

The advantage of the second method is that we do not have to model the way platosim generate the star displacement.

Obviously with the current version of pproc.py, the method 1 cannot take into account the Kinematic aberration, and therefore we will have to use the second method (For the time being the 'drift and jitter' correction algo  uses the file <starID>_displacement.dat: but we can change this).

I've committed a new version of pproc.py which now by default generates star displacement at low cadence (25s -> the second method described above). This way we can manage any kind of displacement (including displacement due to the differential kinetmatic abberation provided that the latter is included in colPix and  rowPix). In that case we ignore/neglect the star displacement occuring at high cadence (e.g. due to the jitter).

As a consequence in invert.py I had to change the value of the -N when calling invert_parabolic1. The new value is 1 because the star displacement is only know at the exposure cadence. The previous value was 168 because the  star displacemen was know at the cadence of 8Hz and 168*1/8 = 21 s, which is integration time.

So you have to update both pproc.py and invert.py

When a small centroid error is made, the (small) differences between the predicted/modeled light-curve and the one obtained from the simulated imagette do increase with time.
